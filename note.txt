

Microservice Transformation คือ
การเปลี่ยนแปลงระบบซอฟต์แวร์จากรูปแบบโมโนลิธิก (Monolithic) ไปสู่รูปแบบไมโครเซอร์วิส (Microservices) ซึ่งเป็นแนวทางในการออกแบบและพัฒนาระบบที่แบ่งออกเป็นส่วนย่อยๆ ที่สามารถทำงานได้อย่างอิสระ 
โดยแต่ละส่วนจะมีหน้าที่เฉพาะเจาะจงและสามารถสื่อสารกันผ่าน API หรือโปรโตคอลต่างๆ ได้อย่างมีประสิทธิภาพ การเปลี่ยนแปลงนี้ช่วยให้ระบบมีความยืดหยุ่นสูงขึ้น สามารถปรับปรุงและขยายได้ง่ายขึ้น และลดความซับซ้อนในการพัฒนาและบำรุงรักษาระบบในระยะยาว
การเปลี่ยนแปลงไปสู่ Microservices มีขั้นตอนหลักๆ ดังนี้:
1. การวิเคราะห์และออกแบบ: ทำความเข้าใจระบบปัจจุบันและกำหนดขอบเขตของแต่ละไมโครเซอร์วิส
2. การแยกส่วน: แบ่งระบบออกเป็นส่วนย่อยๆ ตามฟังก์ชันหรือโดเมนที่เกี่ยวข้อง
3. การพัฒนาและทดสอบ: พัฒนาแต่ละไมโครเซอร์วิสแยกกันและทดสอบอย่างละเอียดเพื่อให้แน่ใจว่าทำงานได้อย่างถูกต้อง
4. การปรับใช้และบำรุงรักษา: นำไมโครเซอร์วิสไปใช้งานจริงและดูแลรักษาอย่างต่อเนื่องเพื่อให้ระบบมีประสิทธิภาพและเสถียรภาพสูงสุด
การเปลี่ยนแปลงไปสู่ Microservices อาจมีความท้าทายในการจัดการกับความซับซ้อนของระบบและการสื่อสารระหว่างไมโครเซอร์วิส
แต่ด้วยการวางแผนและการดำเนินการที่ดี สามารถช่วยให้ระบบมีความยืดหยุ่นและสามารถปรับตัวได้ดีในอนาคต

digital transformation คือ
การเปลี่ยนแปลงทางดิจิทัล (Digital Transformation) 
คือกระบวนการที่องค์กรนำเทคโนโลยีดิจิทัลมาใช้ในการปรับปรุงและเปลี่ยนแปลงวิธีการทำงาน กระบวนการทางธุรกิจ 
และประสบการณ์ของลูกค้า เพื่อให้สามารถแข่งขันได้ในยุคดิจิทัล

Cloud Native คือแนวทางในการพัฒนาและดำเนินการแอปพลิเคชันที่ออกแบบมาเพื่อทำงานในสภาพแวดล้อมของคลาวด์โดยเฉพาะ
แอปพลิเคชันที่เป็น Cloud Native จะถูกสร้างขึ้นโดยใช้เทคโนโลยีและเครื่องมือที่เหมาะสมกับการทำงานในคลาวด์ เช่น คอนเทนเนอร์ (Containers), ไมโครเซอร์วิส (Microservices), และการจัดการแบบอัตโนมัติ (Automation)
การพัฒนาแอปพลิเคชันแบบ Cloud Native ช่วยให้สามารถปรับปรุงและขยายแอปพลิเคชันได้อย่างรวดเร็วและมีประสิทธิภาพในสภาพแวดล้อมของคลาวด์ นอกจากนี้ยังช่วยให้สามารถใช้ประโยชน์จากความสามารถของคลาวด์ เช่น การปรับขนาดอัตโนมัติ (Auto-scaling) และการจัดการทรัพยากรอย่างมีประสิทธิภาพ


# หนังสือ
Cloud Native Transformation เป็นหนังสือที่พูดถึงการเปลี่ยนแปลงองค์กรและกระบวนการพัฒนาแอปพลิเคชันไปสู่แนวทาง Cloud Native
หนังสือเล่มนี้เน้นการนำเสนอแนวทางและกลยุทธ์ในการเปลี่ยนแปลงองค์กรให้สามารถพัฒนาและดำเนินการแอปพลิเคชันในสภาพแวดล้อมของคลาวด์ได้อย่างมีประสิทธิภาพ
เนื้อหาในหนังสือจะครอบคลุมเรื่องการออกแบบแอปพลิเคชันแบบ Cloud Native, การใช้เทคโนโลยีและเครื่องมือที่เหมาะสม, และการจัดการการเปลี่ยนแปลงในองค์กรเพื่อให้สามารถปรับตัวได้ในยุคดิจิทัล


SDLC คือ
SDLC (Software Development Life Cycle) คือกระบวนการที่ใช้ในการพัฒนาซอฟต์แวร์ ซึ่งประกอบด้วยขั้นตอนต่างๆ ที่ช่วยให้การพัฒนาซอฟต์แวร์เป็นไปอย่างมีระบบและมีประสิทธิภาพ
ขั้นตอนหลักของ SDLC ได้แก่:
1. การวางแผน (Planning): กำหนดวัตถุประสงค์และขอบเขตของโครงการ รวมถึงการจัดสรรทรัพยากรและการกำหนดเวลา
2. การวิเคราะห์ความต้องการ (Requirements Analysis): รวบรวมและวิเคราะห์ความต้องการของผู้ใช้และระบบ เพื่อให้เข้าใจถึงสิ่งที่ต้องการจากซอฟต์แวร์
3. การออกแบบ (Design): วางแผนโครงสร้างและสถาปัตยกรรมของซอฟต์แวร์ รวมถึงการออกแบบฐานข้อมูลและอินเทอร์เฟซผู้ใช้
4. การพัฒนา (Development): เขียนโค้ดและสร้างซอฟต์แวร์ตามการออกแบบที่ได้วางไว้
5. การทดสอบ (Testing): ทดสอบซอฟต์แวร์เพื่อให้แน่ใจว่าทำงานได้อย่างถูกต้องและไม่มีข้อผิดพลาด
6. การปรับใช้ (Deployment): นำซอฟต์แวร์ไปใช้งานจริงในสภาพแวดล้อมของผู้ใช้
7. การบำรุงรักษา (Maintenance): ดูแลและปรับปรุงซอฟต์แวร์อย่างต่อเนื่องเพื่อให้สามารถตอบสนองความต้องการของผู้ใช้และแก้ไขปัญหาที่เกิดขึ้นได้อย่างมีประสิทธิภาพ
การใช้ SDLC ช่วยให้การพัฒนาซอฟต์แวร์เป็นไปอย่างมีระบบและสามารถจัดการกับความซับซ้อนได้ดีขึ้น นอกจากนี้ยังช่วยให้สามารถตรวจสอบและปรับปรุงกระบวนการพัฒนาได้อย่างต่อเนื่องเพื่อให้ได้ผลลัพธ์ที่ดีที่สุดสำหรับผู้ใช้และองค์กร










Container D+Runtime คือแนวคิดในการพัฒนาและดำเนินการแอปพลิเคชันที่ใช้คอนเทนเนอร์ (Containers) และรันไทม์ (Runtime) เพื่อให้สามารถทำงานได้อย่างมีประสิทธิภาพในสภาพแวดล้อมของคลาวด์
คอนเทนเนอร์เป็นเทคโนโลยีที่ช่วยให้สามารถแพ็คเกจแอปพลิเคชันและส่วนประกอบที่เกี่ยวข้องทั้งหมดไว้ในหน่วยเดียว ซึ่งช่วยให้สามารถย้ายและปรับใช้แอปพลิเคชันได้อย่างง่ายดาย

CLI คือ 
Command Line Interface (CLI) คืออินเทอร์เฟซที่ให้ผู้ใช้สามารถโต้ตอบกับระบบคอมพิวเตอร์ผ่านทางบรรทัดคำสั่ง
โดยผู้ใช้จะพิมพ์คำสั่งลงในหน้าต่างเทอร์มินัลหรือคอนโซลเพื่อสั่งให้ระบบทำงานตามคำสั่งนั้น



2. ใช้เป็น event-driven architecture
